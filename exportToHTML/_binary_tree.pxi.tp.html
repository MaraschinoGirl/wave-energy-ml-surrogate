28   # KD Tree and Ball Tree
29   # =====================
30   #
31   # The routines here are the core algorithms of the KDTree and BallTree
32   # structures.  If Cython supported polymorphism, we would be able to
33   # create a subclass and derive KDTree and BallTree from it.  Because
34   # polymorphism is not an option, we use this single BinaryTree class
35   # as a literal include to avoid duplicating the entire file.
36   #
37   # A series of functions are implemented in kd_tree.pyx and ball_tree.pyx
38   # which use the information here to calculate the lower and upper bounds
39   # between a node and a point, and between two nodes.  These functions are
40   # used here, and are all that are needed to differentiate between the two
41   # tree types.
42   #
43   # Description of Binary Tree Algorithms
44   # -------------------------------------
45   # A binary tree can be thought of as a collection of nodes.  The top node
46   # contains all the points.  The next level consists of two nodes with half
47   # the points in each, and this continues recursively.  Each node contains
48   # metadata which allow fast computation of distance bounds: in the case of
49   # a ball tree, the metadata is a center and a radius.  In the case of a
50   # KD tree, the metadata is the minimum and maximum bound along each dimension.
51   #
52   # In a typical KD Tree or Ball Tree implementation, the nodes are implemented
53   # as dynamically allocated structures with pointers linking them.  Here we
54   # take a different approach, storing all relevant data in a set of arrays
55   # so that the entire tree object can be saved in a pickle file. For efficiency,
56   # the data can be stored in such a way that explicit pointers are not
57   # necessary: for node data stored at index i, the two child nodes are at
58   # index (2 * i + 1) and (2 * i + 2); the parent node is (i - 1) // 2
59   # (where // indicates integer division).
60   #
61   # The data arrays used here are as follows:
62   #   data : the [n_samples x n_features] array of data from which the tree
63   #          is built
64   #   idx_array : the length n_samples array used to keep track of the indices
65   #          of data within each node.  Each node has values idx_start and
66   #          idx_end: the points within the node are given by (using numpy
67   #          syntax) data[idx_array[idx_start:idx_end]].
68   #   node_data : the length n_nodes array of structures which store the node
69   #          indices, node radii, and leaf information for each node.
70   #   node_bounds : the [* x n_nodes x n_features] array containing the node
71   #          bound information.  For ball tree, the first dimension is 1, and
72   #          each row contains the centroid of the node.  For kd tree, the first
73   #          dimension is 2 and the rows for each point contain the arrays of
74   #          lower bounds and upper bounds in each direction.
75   #
76   # The lack of dynamic allocation means the number of nodes must be computed
77   # before the building of the tree. This can be done assuming the points are
78   # divided equally between child nodes at each step; although this removes
79   # some flexibility in tree creation, it ensures a balanced tree and ensures
80   # that the number of nodes required can be computed beforehand.  Given a
81   # specified leaf_size (the minimum number of points in any node), it is
82   # possible to show that a balanced tree will have
83   #
84   #     n_levels = 1 + max(0, floor(log2((n_samples - 1) / leaf_size)))
85   #
86   # in order to satisfy
87   #
88   #     leaf_size <= min(n_points) <= 2 * leaf_size
89   #
90   # with the exception of the special case where n_samples < leaf_size.
91   # for a given number of levels, the number of nodes in the tree is given by
92   #
93   #     n_nodes = 2 ** n_levels - 1
94   #
95   # both these results can be straightforwardly shown by induction.  The
96   # following code uses these values in the construction of the tree.
97   #
98   # Distance Metrics
99   # ----------------
100  # For flexibility, the trees can be built using a variety of distance metrics.
101  # The metrics are described in the DistanceMetric class: the standard
102  # Euclidean distance is the default, and is inlined to be faster than other
103  # metrics.  In addition, each metric defines both a distance and a
104  # "reduced distance", which is often faster to compute, and is therefore
105  # used in the query architecture whenever possible. (For example, in the
106  # case of the standard Euclidean distance, the reduced distance is the
107  # squared-distance).
108  #

441  cdef float64_t logVn(intp_t n):

451  cdef float64_t _log_kernel_norm(float64_t h, intp_t d,
452                                  KernelType kernel) except -1:
453      """Given a KernelType enumeration, compute the kernel normalization.
454  
455      h is the bandwidth, d is the dimension.
456      """
457      cdef float64_t tmp, factor = 0
458      cdef intp_t k
459      if kernel == GAUSSIAN_KERNEL:
460          factor = 0.5 * d * LOG_2PI
461      elif kernel == TOPHAT_KERNEL:
462          factor = logVn(d)
463      elif kernel == EPANECHNIKOV_KERNEL:
464          factor = logVn(d) + log(2. / (d + 2.))
465      elif kernel == EXPONENTIAL_KERNEL:
466          factor = logSn(d - 1) + lgamma(d)
467      elif kernel == LINEAR_KERNEL:
468          factor = logVn(d) - log(d + 1.)
469      elif kernel == COSINE_KERNEL:
470          # this is derived from a chain rule integration
471          factor = 0
472          tmp = 2. / PI
473          for k in range(1, d + 1, 2):
474              factor += tmp
475              tmp *= -(d - k) * (d - k - 1) * (2. / PI) ** 2
476          factor = log(factor) + logSn(d - 1)
477      else:
478          raise ValueError("Kernel code not recognized")
479      return -factor - d * log(h)

484

486      ----------
487      h : float

497          kernel norm.

500      knorm or log_knorm : float
501          the kernel norm or logarithm of the kernel norm.

503      if kernel == 'gaussian':
